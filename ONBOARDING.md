# AI編集部 オンボーディング

アトミックソフトウェア AI編集部へようこそ。
このガイドに沿って初回セットアップを済ませると、企画から原稿生成までをAIエージェントに任せられるようになります。

---

## この編集部でできること

音声取材の文字起こしから、企画書と原稿の生成までを4羽のエージェントが担当します。

| エージェント | 役割 |
|---|---|
| カワセミ（企画担当） | 企画テンプレから企画書を生成する |
| ハシボソカラス（ライティング担当） | 企画書と文字起こしから原稿を生成する |
| オオタカ（校正担当） | 表記・構成・内容をチェックし修正する |
| シジュウカラ（最終レビュー担当） | 最終確認とフィードバックの学習反映を行う |

---

## 2つのロールと権限

| | 管理者（高根） | ユーザ（あなた） |
|---|---|---|
| GitHub | 編集・PR承認・マージ | 閲覧・PR作成 |
| Cowork / Claude Code | 利用可 | 利用可 |
| エージェント・スキルの改変 | できる | PR提案まで（マージは高根） |

エージェントの品質基準（スキルファイル）を実際に書き換えられるのは管理者だけです。
あなたのフィードバックはPR（修正提案）として管理者に届き、承認されると全員に反映されます。

---

## ① 初回セットアップ（一度だけ）

### STEP1: GitHub権限をリクエストする

編集部リポジトリ（`Takanechis/cowork-agents`）の「閲覧＋PR作成」権限を高根に申請してください。
申請時はあなたのGitHubアカウント名を伝えてください。高根がコラボレーターに追加します。

### STEP2: エージェントをFetchする

ターミナルで以下を1行実行します。

```bash
curl -sSf https://raw.githubusercontent.com/Takanechis/cowork-agents/main/setup.sh | bash
```

自動で以下が行われます。

- リポジトリを `~/cowork-agents` にクローン
- エージェントを `~/.claude/agents/` にリンク
- 起動時に `git pull` で自動同期するフックを設定

完了後、Claude Code を再起動してください。以降は起動のたびに最新のエージェントへ自動更新されます。

### STEP3: GitHub MCP を接続する

フィードバックを修正提案（PR）として送れるように、GitHubコネクタを接続します。

1. Cowork または Claude Code の設定からコネクタ一覧を開く
2. GitHub を選んで接続する
3. OAuth認証であなたのGitHubアカウントを承認する（スコープは `repo` を許可）

接続後、シジュウカラ（最終レビュー担当）があなたのアカウントで自動的にPRを作成できるようになります。
PRの承認とマージは高根が行います。

### STEP4: 動作確認

Claude Code で次を実行し、エージェントが見えれば成功です。

```
@カワセミ 企画確認
```

## ② 運用マニュアル

あなたがエージェントに渡す情報は2種類です。

- 企画（これから記事のテーマを決める）
- 文字起こしテキスト（取材済みの素材から原稿を作る）

### フローA: 企画

```
1. カワセミに「企画」と伝える
2. カワセミが企画テンプレ（5項目）を返す
3. テンプレを埋めて送り返す
   ・タイトル
   ・テーマ
   ・読者（ターゲット）
   ・読者の読後感・便益
   ・ポイントとなる要素
4. カワセミが企画書を生成し、共有ドライブに案件フォルダと企画書Docを作成してURLを返す
5. 企画書を確認する
   - 問題なければ「OK」と返す → 完了
   - 修正したい点があればフィードバックを入力する → フローC へ
```

### フローB: 文字起こしテキスト → 原稿

```
1. 文字起こしDocと企画書をエージェントに共有する
2. ハシボソカラスが原稿を生成し、オオタカが校正する
3. 共有ドライブに案件フォルダと原稿Docを作成してURLを返す
4. 原稿を確認する
   - 問題なければ「OK」と返す → 完了
   - 修正したい点があればフィードバックを入力する → フローC へ
```

### フローC: フィードバックが学習になる仕組み

```
あなたがフィードバックを入力する
   ↓
シジュウカラがスキルファイルの修正案を生成する
   ↓
GitHub MCP で修正提案のPRを作成する（あなたのアカウントで）
   ↓
高根がPRをレビューする（承認者は高根のみ）
   ↓ マージ
全ユーザに git pull で自動反映される
```

あなたのフィードバックが、編集部全体の品質基準を育てます。

## 取材データの自動取り込み（Slack + Zapier）

取材データをSlackにアップすると、原稿生成まで自動で進みます。
アップしたファイルの拡張子で処理が自動的に分岐します。

| 拡張子 | 種別 | 処理 |
|---|---|---|
| .m4a / .mp3 / .wav / .aac / .ogg / .flac / .wma / .mp4 | 音声 | 文字起こししてから共有ドライブへ配置 |
| .txt / .md / .pdf / .docx / .doc / .rtf | テキスト | そのまま共有ドライブへ配置 |

```
[取材] 各自がローカルで録音、または文字起こし
   ↓
[共有] Slackにファイルをアップ（スレッドに案件IDを記載。例: 2026-001）
   ↓ Zapier「Slack新規ファイル」トリガー
   ↓ 拡張子で分岐
   ├─ 音声 → 文字起こし（Whisper等）→ 共有ドライブの案件フォルダ
   └─ テキスト → そのまま共有ドライブの案件フォルダ
   ↓ Cowork検知
[原稿生成] ハシボソカラス → オオタカ → 原稿Doc
   ↓ 通知
[レビュー] 原稿を確認する → OK または FB
```

案件への紐付けは、Slackにアップする際にスレッドまたは投稿本文へ案件ID（例: 2026-001）を記載することで行います。Zapierがその案件IDを読み、共有ドライブの該当フォルダへ配置します。

### Zapier設定（管理者向け）

- トリガー: Slack → New File（対象チャンネル）
- 分岐: Paths by Zapier でファイル名の拡張子を判定する
  - 音声系の拡張子 → Transcription by Zapier または Whisper でテキスト化 → Google Drive の案件フォルダにアップ
  - テキスト系の拡張子 → Google Drive の案件フォルダにそのままアップ
- 配置先フォルダは、投稿に記載された案件IDから特定する

## 困ったとき

- エージェントが見えない → Claude Code を再起動する（STEP2のフックで自動同期されます）
- 権限エラーが出る → STEP1のGitHub権限とSTEP3のGitHub MCP接続を確認する
- 表記や品質の指摘がある → 遠慮なくフィードバックを入力する。PRとして高根に届きます

ミッション「未来に、もっと余白を。」を、編集業務からも実現していきましょう。
