# regulation-md-writing.md

## MD ライティング・レギュレーション

このファイルは、cowork-agents リポジトリ内の `.md` ファイルを記述する際に守るべきルールを定める。
AIエージェントおよびコントリビューターは、本ファイルのルールに従って Markdown を記述すること。

---

## 設計思想

### 学習ループの構築

このリポジトリのスキル・エージェント設計の核は「学習のループが回る仕組み」にある。業務プロセスを洗い出し、作業順序の基盤を用意するだけでなく、レビューを繰り返すことでモデルが賢くなっていく循環を自動化することが目的である。

MD ファイルは単なる手順書ではなく、蓄積と進化のための器として設計する。

### エッセンス抽出と蓄積

レビューで得られた指摘・改善点はすべてエッセンス候補となる。ただし、エッセンスとして MD に組み込むかどうかは人間がジャッジする。承認（プルーブ）されたものだけが正式にルールとして反映される。

フロー:

1. エージェントが原稿・成果物を生成する
2. 人間がレビューし、指摘・修正を行う
3. 指摘内容からエッセンス候補を抽出する
4. 人間が「これはルールとして定着させるべきか」を判断する
5. 承認されたエッセンスのみ、該当する MD ファイルに反映する

### MD 設計の原則

- 業務にアジャストしたスキルモデルをまず作り、そこにエッセンスを蓄積していく
- スキルやエージェントは「学習サイクルが勝手に回る仕組み」を前提に設計する
- レビュアーの指摘が次回以降の生成品質に自動的に活かされる構造にする

## Markdown フォーマットルール

### 1. インデントとスペース

- **タブ文字（\t）の使用を禁止する。** インデントは半角スペース 2 つを単位とする。
- 行頭に不要なスペースを入れない。リストや本文が意図せずインデントされた状態で記述されることを禁止する。
- ネストされたリストは、半角スペース 2 つのみでインデントする。

### 2. 引用記法（>）の使用禁止・制限

- `>` による blockquote 記法を、注記・補足・ルール説明などの通常コンテンツに使用することを禁止する。
- `> > >` のような多段ネストの blockquote は絶対に使用しない。
- 補足・注記を記述する場合は、通常のリスト（`-`）またはセクション見出し（`###`）を使用する。

悪い例（禁止）:

```
> **抽出ルール補足**
> > - キーワードはファイル名の先頭・中間・末尾どこに現れても抽出対象とする
> > > - 区切り文字の有無に関わらずマッチする
```

良い例:

```
### 抽出ルール補足
- キーワードはファイル名の先頭・中間・末尾どこに現れても抽出対象とする
- 区切り文字（\`_\`・\`-\`・スペース）の有無に関わらずマッチする
```

### 3. リスト番号の重複・連番ミス禁止

- 番号付きリスト（1., 2., ...）において、同じ番号を重複して記述することを禁止する。
- 番号付きリストのネストは、2 スペースのインデント＋ `1.` 形式で記述する。

悪い例（禁止）:

```
1. フォルダ URL にアクセスする
2. 2. 10ティック × 8回 スクロールダウンする
3. 3. ドキュメントタイトルを抽出する
```

良い例:

```
1. フォルダ URL にアクセスする
2. 10ティック × 8回 スクロールダウンする
3. ドキュメントタイトルを抽出する
```

### 4. リストマーカーの重複禁止

- リスト項目の行頭に `- -` のように同じマーカーを二重で記述することを禁止する。

悪い例（禁止）:

```
- - \`注文書\`
- - \`顧問\`
```

良い例:

```
- \`注文書\`
- \`顧問\`
```

### 5. セクション区切り（---）の使い方

- `---` による水平線は、**大きなセクションの区切りにのみ**使用する。
- リスト項目の中や blockquote の中に `---` を埋め込まない。
- 連続する `---` の多用は避け、1 ファイルにつき必要最小限にとどめる。

### 6. 見出しレベルの整合性

- ファイル全体で見出しレベル（`#`, `##`, `###`, ...）は論理的な階層構造を保つ。
- `##` の子見出しは `###` とし、`##` から `####` へ直接飛ぶことを禁止する。
- Step・フェーズの区切りには `## Step N：タイトル` 形式を使用する。

### 7. 全体的な可読性ルール

- 1 行あたりの文字数は原則 120 文字以内を目安とする（長い URL は除く）。
- 意味のある空行を使い、セクション間・リスト間に 1 行の空行を入れる。
- 1 つの箇条書き項目に複数の意味を詰め込まない。1 項目 1 情報の原則を守る。
- コードやコマンドはインラインコード（` `）またはコードブロック（```）で明示する。

### 8. フロントマター（YAML）ルール

- `---` で囲まれたフロントマターは、ファイル先頭にのみ記述する。
- フロントマター内のフィールドはインデントなし、1 フィールド 1 行で記述する。

## 原稿ライティングのトンマナルール

レビューから抽出されたエッセンスを以下にまとめる。エージェントが原稿を生成する際、このルールに従うこと。

### トーンの原則

- ホスピタリティ（寄り添う気持ち）は保ちつつ、ロジカルに解決に導く文体にする
- 甘くするのではなく「余計なことを加えず、論理的に導く」姿勢
- 冒頭から攻撃的・強い印象を与える表現を避ける

### 表現ルール

- 「させていただく」の過剰使用を禁止する
- 根拠なき修飾語（画期的な、革新的な 等）を排除する

### 構成ルール

- 機能説明は文脈なしに始めない。状況を示してから説明に入る（例: 「在庫の数がシステムと合いません」という状況を先に示し、そこからどう解決するかを説明する）
- 本筋と関係のない内容は削除する。関連性が薄いと判断した段落は思い切って外す
- メンバー紹介では、すでに十分伝わっている内容の繰り返しを避ける
- 「ワンチーム」として語る場合、個人の組み合わせを無理に強調しない

### 個人情報・機密の扱い

- 記事に登場する人物の名前はマスキングする
- 日付やイベント情報から個人が特定される可能性がある場合、ぼかす
- 詳細は `skills/manuscript-writing/information-disclosure-checklist.md` を参照

## このレギュレーションが禁止するノイズ一覧

| ノイズパターン | 禁止理由 |
| --- | --- |
| タブ文字によるインデント | エディタ依存で表示が崩れる |
| `> > >` 多段 blockquote | 可読性が著しく低下する |
| `- -` 二重リストマーカー | 構文的に不正・見た目が混乱する |
| `2. 2.` 番号の重複 | 番号リストの意味が失われる |
| リスト内の `---` | セクション区切りとして機能しない |
| 不要な先頭スペースによるインデント | 意図しないコードブロック化が起こる |
| 文脈なしの機能説明 | 読者に伝わらない |
| 本筋と無関係な段落 | 原稿の焦点がぼやける |
