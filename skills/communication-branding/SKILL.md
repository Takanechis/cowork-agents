# Communication & Branding スキル定義

> ベンチマーク: SmartHR Design System（https://smarthr.design/）
> 教師データ: 「ちいさくはじめるデザインシステム」（SmartHR社）
> 対象: アトミックソフトウェア株式会社のデザインシステム構築

---

## 教師データから学んだ原則

### デザインシステムとは
- UIコンポーネントやスタイルガイドだけではない
- プロダクトとブランドコミュニケーション両方をカバーする統合的な基準
- 「だれでも・効率よく・迷わずに」ブランド表現ができるようにする仕組み

### 誰のためのデザインシステムか
1. プロダクト開発チーム: 一貫性のあるUI設計、開発効率化
2. ブランドコミュニケーション: マーケ・セールス・CS等が使う素材の品質担保
3. 全社メンバー: デザイナーだけでなく、エンジニア・PM・ビジネスサイドも利用者

### ちいさくはじめる3つのステップ
1. STEP1: 最小限の範囲で始める（全部を一度に作らない）
2. STEP2: 日常業務の中で課題を発見し、都度ルール化する
3. STEP3: コンテンツの作者になってもらう（巻き込み）

### みんなのものにする3つのステップ
1. STEP1: 巻き込む仲間を見つける
2. STEP2: デザインシステムの意義を理解してもらう
3. STEP3: コンテンツの作者になってもらう

### 運用の鍵
- 専任チームがなくても回せる設計にする
- 業務を通じた課題発見と解決のサイクルを回す
- フィードバックを常に受け付ける仕組みを作る
- 効果検証と大きな更新はアンケートやプロジェクト化する

## SmartHR Design System 構造マップ

### Tier 1: 基本原則（なぜ）
- パーソナリティ: ブランドの4つの価値観
- デザイン原則: 設計判断の基準
- 運営理念: デザインシステム自体の目的と方針

### Tier 2: 基本要素（何を使うか）
- ロゴ: 使用規定・クリアスペース・最小サイズ・禁止事項
- 色: プライマリ/セカンティック/グレースケール + セマンティックカラー
- タイポグラフィ: 書体・サイズスケール・行間・ウェイト
- アイコン: アイコンセット・グリッド・使用ガイドライン
- イラストレーション: スタイル・トーン・使用場面

### Tier 3: デザイントークン（どう管理するか）
- 色トークン・タイポグラフィトークン・スペーシングトークン
- JSONまたはCSS Custom Propertiesで管理
- プロダクトとドキュメントで同じトークンを参照する

### Tier 4: コンポーネント（どう組み立てるか）
- ボタン / 表示 / フォーム / レイアウト / ナビゲーション / オーバーレイ / 状態 / テキスト

### Tier 5: パターン（どう使うか）
- デザインパターン: よくある画面構成のテンプレート
- ライティング: UIテキスト・エラーメッセージ・用字用語

### Tier 6: コミュニケーション（社外にどう伝えるか）
- 写真・動画 / アイキャッチ / スライド・資料 / 名刺 / 画面キャプチャ

### Tier 7: アクセシビリティ（誰もが使えるか）
- WCAG 2.1 AA準拠 / チェックリスト / 代替テキスト / 多言語化

## ロードマップ: アトミックソフトウェア デザインシステム構築

### Phase 0: 準備（1〜2週間）
目的: 現状把握と方針決定

- [ ] 既存のデザイン資産の棚卸し
  - ロゴデータ（medicalforce / 警備フォース / アトミックソフトウェア）
  - 現在使用しているカラーパレット
  - 使用フォント
  - 既存のスライドテンプレート・名刺・資料
- [ ] 現状の課題リストアップ
  - ブランド表現の不統一箇所
  - 制作物で毎回迷うポイント
  - チーム間での認識のズレ
- [ ] デザインシステムのスコープ決定
  - まずカバーする領域（プロダクト？コミュニケーション？両方？）
  - 最初に着手する優先順位

### Phase 1: 基盤構築（2〜4週間）
目的: ブランドの「なぜ」と「基本素材」を定義する

- [ ] パーソナリティの定義
  - アトミックソフトウェアの価値観を3〜4個の形容詞で定義
  - 各パーソナリティの「する/しない」リスト作成
- [ ] デザイン原則の策定
  - 設計判断に迷ったときの基準を3〜5個定義
- [ ] ロゴ使用規定の文書化
  - クリアスペース・最小サイズ・背景色ルール・禁止事項
  - medicalforce / 警備フォース / アトミックソフトウェアの3ブランド
- [ ] カラーパレットの定義
  - プライマリ・セカンダリ・アクセント・グレースケール
  - セマンティックカラー（成功/警告/エラー/情報）
  - コントラスト比の確認（WCAG AA基準）
- [ ] タイポグラフィの定義
  - 見出し・本文・キャプションのサイズスケール
  - 行間・字間の基準値

### Phase 2: トークン化とコンポーネント（4〜6週間）
目的: デザイン決定を再利用可能な形にする

- [ ] デザイントークンの定義
  - 色トークン（プリミティブ → セマンティック）
  - タイポグラフィトークン
  - スペーシングトークン
  - JSON/CSS Custom Properties形式で管理
- [ ] コンポーネントライブラリの構築開始
  - ボタン（プライマリ/セカンダリ/テキスト/アイコン付き）
  - フォーム要素（テキストフィールド/セレクト/チェックボックス/ラジオ）
  - カード/リスト/テーブル
  - ナビゲーション/ヘッダー/フッター
- [ ] 文字コンテンツガイドラインの策定
  - 用字用語リスト（固有名詞表記統一）
  - UIテキストの書き方ルール
  - エラーメッセージのテンプレート

### Phase 3: コミュニケーション素材（2〜4週間）
目的: 社外向けのブランド表現を統一する

- [ ] 写真・動画のスタイルガイド
  - トーン・フィルター・構図の基準
- [ ] スライド・資料テンプレート
  - PowerPoint/Googleスライドのマスターテンプレート
  - 表紙・目次・コンテンツ・まとめのレイアウト
- [ ] 名刺デザイン規定
- [ ] 画面キャプチャ・端末モックのガイドライン
- [ ] アイキャッチ画像のテンプレート

### Phase 4: ドキュメント化と展開（2〜3週間）
目的: 誰もが参照・貢献できる状態にする

- [ ] デザインシステムドキュメントサイトの構築
  - SmartHR Design System（smarthr.design）のような公開サイト
  - または社内Notion/Confluenceでの公開
- [ ] 社内オンボーディング
  - 使い方ガイドの作成
  - キックオフ共有会の開催
- [ ] フィードバック受付の仕組み
  - Slack チャンネル / GitHub Issues / Notion フォーム

### Phase 5: 運用と継続改善（継続）
目的: 生きたデザインシステムとして進化させる

- [ ] 定期レビューサイクルの確立（月次or四半期）
- [ ] 更新ログの記録
- [ ] 利用状況のモニタリング
- [ ] 新規コンポーネント追加のフロー整備
- [ ] アクセシビリティ監査の定期実施
