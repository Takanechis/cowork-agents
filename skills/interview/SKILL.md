# skills/interview/SKILL.md — インタビュー企画・原稿共通スキル

このファイルは、インタビュー企画（@interview-writer）およびレビュー（@interview-reviewer）エージェントが参照する共通ルール・品質基準を定める。

---

## 対象媒体

- VerticalSaaS Mag.（外部向けインタビュー記事）
- アトミックソフトウェア オウンドメディア note（社内インタビュー記事）

---

## 5大ニュース価値

企画は以下のうち最低 2 つを満たすこと。

| 価値 | 問い | 判定基準 |
|---|---|---|
| 新規性 | 業界で初めての取り組みか | 「業界初」「日本初」と言える要素があるか |
| 意外性 | 読者の予想を裏切る要素があるか | 「まさかあの企業が」と思わせる切り口か |
| No.1 | 何かの領域で突出した実績があるか | 市場シェア・導入数・成長率など数字で語れるか |
| 時事性 | 今このタイミングで出す意味があるか | 法改正・業界トレンド・資金調達直後など |
| 権威性 | 信頼できる第三者からの評価があるか | 受賞歴・著名投資家・業界団体での役職など |

判定基準: ◎ 2つ以上 → Go / ◎ 1つかつ ○ 2つ以上 → 条件付き Go / 上記未満 → Rework

---

## 読者 Give の基準

| Give の種類 | 説明 |
|---|---|
| 判断軸 | 意思決定に使えるフレームワークや考え方 |
| 数字 | ベンチマークになる具体的な指標 |
| 失敗談 | 追体験できる具体的な失敗と学び |
| HOW | 再現可能な手順・プロセス |
| 視座 | 業界全体を俯瞰する洞察 |

判定基準: ◎ 2つ以上 → Go / ◎ 1つかつ ○ 2つ以上 → 条件付き Go / 上記未満 → Rework

---

## 企画書フォーマット（標準構成）

1. タイトル案（2〜3パターン）
2. 企画概要（2〜3文）
3. 企画の目的
4. 想定読者（プライマリ / セカンダリ）
5. 読後感
6. インタビュー対象者（氏名・役職・経歴サマリ）
7. 構成案（前提知識の補足 + テーマ1〜2 + 各質問 Q1〜Q5）
8. サマリー
9. 参考リンク（個別記事の正確なURLのみ）

---

## ライティングルール

- テーマは 1〜2 本に絞る。各テーマへの深堀りを十分に行う
- 質問は「Why / How / 具体的には」を軸に設計し、Yes/No で終わる質問は避ける
- 固有名詞・数字を必ず含める。抽象論に終始しない
- 前提知識の補足は必ず作成する
- 推測や仮説は（推測）（要確認）とラベリングする
- 参考リンクには必ず個別記事の正確なURLを記載する

---

## 禁止表現

- 「どうでしたか？」のような漠然とした質問
- 公開情報で分かることをそのまま質問する（「御社の事業内容を教えてください」等）
- 取材相手に先入観を押し付ける質問（「〜に苦労されたと思いますが」等）
- 競合を直接的に比較する質問
- 質問項目で「？」を二つ並べる
- 見出しに「——」を使って語句をつなげる

---

## 品質ゲート（出力前チェック）

1. テーマの質問が具体的か
2. テーマが分散していないか
3. 既存記事と切り口が被っていないか
4. 読者価値として持ち帰れる学びが明確か
5. 固有名詞や数字が入っているか
6. 構成に流れがあるか
7. 参考リンクが 3 つ以上あり、すべて個別記事の正確なURLか
8. 企画概要だけ読んで取材依頼メールが書けるか
9. 前提知識の補足が十分か
10. 各テーマに十分な深堀りができる質問設計になっているか
