# ライティングスキル: オウンドメディア エッセイ記事

## Overview

対象媒体: note（オウンドメディア）
読者: 同じ職種・課題を抱える実務担当者（PR・採用広報・コンテンツ制作等）
目的: 書き手自身の実体験・知見・思想を通じて、読者が実践に持ち帰れる知見を届ける
文体: 「です・ます」調

---

## ライティング原則

- 実体験ベースで書く。推測や一般論よりも「自分がやったこと」「実際に起きたこと」を中心に据える
- 読者が「自分でもできそう」と感じられる解像度で書く。方法論・手順・数字を惜しまない
- 問題意識（なぜ必要か）→ 気づき・設計（どう解決したか）→ 実践・変化（どうなったか）の流れで構成する
- 抽象概念を先に示し、具体事例で後から補強する（演繹的な構成）
- 抽象的な概念や用語は、説明だけで終えず必ず具体例（数字や固有の事象、×と◯の対比など）を添える。例がないまま抽象論を続けない
- 話の粒度が異なるエピソードが混在する場合、大きい概念（方針・全体像）を先に、個別事象（特定部門の話・細かい実装例）を後に配置する
- 「後述する」と書いたら、必ず後で説明する。宣言した情報を回収しない文章を禁止する
- 固有名詞・専門用語は初出時に補足説明を入れる。読者が知らない前提で書く
- 段落ごとに文字数を均一にしない。各段落には必要な情報を網羅的に含め、情報量に応じて文字数が自然に変動することを許容する

## 構成テンプレート

### タイトル

- 引用フレーズ（書き手の言葉・象徴的なフレーズ）＋ サブテーマの二部構成を基本とする。note では短い単独タイトルも可とし、サブテーマを無理に足さない
- 引用フレーズは「機能・手法・具体的な状況」より「書き手の姿勢・気づき・価値観」を表す言葉を選ぶ
- サブテーマは「〜の実像」（現状説明）より「〜が変わった」「〜のあり方」（変化・方向性）を表す語を使う
- タイトルに「：」を使わない

### リード文（2〜3段落）

1. 書き手の問題意識または時事的な文脈の提示（第1段落末尾は体言止め可）
2. 記事で扱うテーマへの橋渡し（「〜はどう変わるのか」等の問いを添えてもよい）
3. 記事で共有する内容の予告（1〜2文）

- リード文は「です・ます」調で記述する
- 第1段落の末尾は体言止めを使ってよい（リズムを出す）
- 「記事の予告文」は結論を開示しすぎない。「この記事では〜を共有します」程度にとどめる
- 「〜の言葉は〜展開されます」等の原稿予告文は入れない

### 本文

- 中見出し: 3〜6個
- 中見出しは「そのセクションを読むと何が分かるか（洞察・結論）」を込めた示唆型を基本とする。素朴な問いかけ型（例、なぜ〜するのか？）も可とする
- 中見出しにトピック説明型（「〜について」「〜とは何か」「〜の概要」）を使わない
- 中見出しに「：」を入れない
- 各段落の冒頭に論理接続語を置き、前段落の論理を受けて次段落を開始する
- 概念説明セクションでは、「番号 ＋ 名称 ＋ 説明」の形式で列挙してよい
- 概念の数を見出しで「4つ」「3つ」と宣言した場合、本文中の個数と一致させる
- 対比・対照を示す例（英語と日本語、翻訳調と自然な日本語、✗と◯など）は、地の文に詰め込まず箇条書きで示し、読者が一目で違いをつかめるようにする。各行にラベル（英語／日本語、✗／◯ など）を付ける

### 末尾

- 実践への呼びかけ、または前向きな展望・意志表明で締める
- 「〜の本質だと考えています」等の結論宣言より「〜から始められます」「〜を実践していきます」等のコミットメントを優先する
- 「ーーありがとうございました。」は使わない（インタビュー記事固有の締め）

## 文体

- 「です・ます」調を基本とする
- 一人称「私」または「私たち」は積極的に使ってよい（実体験ベースの記事のため）
- 同じ語尾が3回以上続かないようにする
- 体言止めの連発を禁止する（第1段落末尾のみ許可）
- 1文は40〜80文字を目安とし、長文は分割する
- 主語と述語の距離を近く保つ
- 根拠を示した上で断定する。「〜と思われる」「〜と考えられる」の多用を禁止する
- 因果関係を明示する（「〜のため」「〜によって」「〜の結果」を適切に使用する）

### 語り口・距離感（note エッセイ）

note のエッセイは、整った論説より「書き手が普段の言葉で語っている」距離感を優先する。整いすぎた文章はかえってAIっぽく見えるため、次を守る。

- 硬い論説調を避け、普段の言葉で素直に書く。ただし口語マーカー（なー、めちゃくちゃ、なんか 等）を盛らない。多用するとかえって嘘くさく、あざとくなる。素の声に近い淡々とした文体を目指す
- 文末の「〜ね。」（「〜ですね。」「〜ですよね。」を含む）は多用しない。1記事に1つまでとする。とくに「〜わけですね。」は使わない
- 書き手の主観や実感を前に出す。一人称の素直な感想で書き出してよい（例、〜と思っています、〜はありがたい）
- 凝った対句や畳みかけを多用しない。「少し前まで〜。今は違います」「Aではなく、B」のような作り込んだ言い回しは、1記事に多くても1回までにする
- 言い切りや体言止めを連ねて文章を「決めすぎ」ない。クリアに整えるより、息づかいが残るほうを優先する
- モノを主語にした受け身より、人を主語にした能動で書く（例、×文章は読み止められます → ◯読むのをやめる人も多い）
- 読者への問いかけや推量でやわらげてよい（例、〜人も多いのではないでしょうか）。語尾に変化をつける範囲で使い、多用はしない

## 禁止表現（エッセイ固有）

regulation-notation.md の共通禁止表現に加えて、以下を禁止する。

- 根拠のない断定（「間違いなく」「必ず」「絶対に」等）
- 曖昧な主語による無根拠な一般化（「人々は」「社会は」「現代では」等）
- 感情的な誇張（「驚くべき」「信じられないほど」等）
- 「後述する」と書いた内容を後で説明しないこと
- 見出しで宣言した数（「4つ」「3つ」等）と本文の列挙数が合わないこと
- 構成を実況・予告する宣言文を書かない（例、「これから結論を言います」「ここからは〜を説明します」「まとめると」）。前ふりせず中身から書く
- 読点「、」を打ちすぎない。息継ぎやリズム調整では打たず、主題や意味の切り替わりにだけ打つ（regulation-notation.md・japanese-style.md の文章のリズムに従う）

## 品質ゲート

以下をすべて満たしていれば Go。1つでも NG なら Rework。

- タイトルに「：」が含まれていない
- リード文が2〜3段落で書かれている
- 中見出しが示唆型または問いかけ型になっている（トピック説明型でない）
- 「後述する」と書いた内容が後で必ず説明されている
- 見出しで宣言した概念の数と本文の列挙数が一致している
- 初出の固有名詞・専門用語に補足説明がある
- 末尾がコミットメントまたは前向きな展望で締められている
- 禁止表現が0件（regulation-notation.md およびエッセイ固有の禁止表現）

## バリアント: Tips Ver.（Zenn「やってみた」スタイル）

Standard Ver. のルールをすべて継承した上で、以下を上書き・追加する。

### Overview（Tips Ver.）

対象媒体: Zenn（技術ブログ）
読者: エンジニア＋業務改善に関心のあるビジネス職のハイブリッド層
目的: 実装・設計の知見を再現可能な形で共有する。読者が「自分でも作れる」と感じる解像度を最優先にする
文体: 技術的だがカジュアル。「やってみた」「詰まった」「工夫した」を積極的に使う

### 構成テンプレート（Tips Ver.）

#### TL;DR（冒頭必須）

記事の最初に必ず置く。以下を箇条書き3〜5行で収める。

- 何を作ったか（1行）
- 使った技術スタック（1行）
- 一番の工夫または学び（1行）
- GitHubリポジトリ（あれば）

#### タイトル（Tips Ver.）

- Standard Ver. の引用フレーズ＋サブテーマ構成でも可
- 「〜をやってみた」「〜を設計した話」「〜を作ったので全部公開する」等の技術ブログ慣用タイトルも許可
- タイトルに「：」は使わない（Standard Ver. と同じ）

#### 本文構成（Tips Ver.）

以下のセクション順を基本とする。

1. **背景と課題**（なぜ作ったか。Standard Ver. のリード文に相当）
2. **システム全体像**（アーキテクチャ図またはファイル構造のコードブロック）
3. **実装①②③…**（セクションを実装単位で分割。番号付き見出しを使ってよい）
4. **工夫した点・詰まった点**（実装上の判断・失敗・トレードオフを可視化する）
5. **セットアップ方法**（読者が再現できるコマンドや手順）
6. **まとめと今後**（コミットメントまたは展望で締める）

#### 見出し（Tips Ver.）

- Standard Ver. の示唆型を基本とするが、「## 実装①：〇〇の定義」等の番号付き実装見出しを許可する
- 実装見出しのみ「：」を使ってよい（番号と内容を区切る用途に限定）

#### コードブロック・構造図（Tips Ver. 追加ルール）

- ファイルパス・ディレクトリ構造・設定ファイル・コマンドは必ずコードブロックで示す
- アーキテクチャや関係性はテキストの構造図（`├─` 等）またはコードブロックで表現する
- 実際のファイル内容を抜粋して示す。架空の例を作らない
- コードブロックには言語タグを付ける（`\`\`\`yaml`、`\`\`\`bash` 等）

#### 「工夫した点・詰まった点」セクション（Tips Ver. 追加ルール）

- 実装上の判断・失敗・トレードオフを正直に書く
- 「〜という問題があった」→「〜で解決した」の構造で1項目ずつ書く
- 「詰まった」は具体的に書く。「うまくいかなかった」で終わらず、どう解決したかまで書く

### 品質ゲート（Tips Ver.）

Standard Ver. の品質ゲートに加えて、以下をすべて満たすこと。

- TL;DR が冒頭にある
- システム全体像またはファイル構造のコードブロックがある
- 実装セクションに実際のコード・設定・コマンドの例がある
- 「工夫した点・詰まった点」セクションがある
- セットアップ方法または再現手順がある
